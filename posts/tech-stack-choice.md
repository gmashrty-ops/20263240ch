# 프론트엔드 프레임워크, 무엇을 선택할까? 🌐

웹 개발을 처음 시작하면서 HTML과 CSS, 그리고 바닐라 자바스크립트로 작은 토이 프로젝트들을 만들었습니다. 하지만 점점 애플리케이션의 규모가 커지고 상태 관리가 복잡해지면서, 가장 먼저 마주친 고민은 **"어떤 최신 도구와 프레임워크를 사용할 것인가?"**였습니다. 

웹 생태계에는 React, Vue, Svelte 등 수많은 선택지가 있었지만, 오랜 고민 끝에 저는 **React**를 기반으로 한 **Next.js**를 주력 기술 스택으로 선택했습니다. 이 글에서는 제가 왜 이런 결정을 내렸는지 그 과정을 기록해 둡니다.

---

## 1. 컴포넌트 기반 UI, React

가장 먼저 React를 선택한 이유는 재사용 가능한 컴포넌트 단위로 개발하는 방식이 너무나 매력적이었기 때문입니다. 이전에는 HTML 파일 하나에 수많은 코드가 섞여 유지보수가 힘들었지만, React의 JSX와 함수형 컴포넌트는 UI를 레고 블록 조립하듯 만들 수 있게 해주었습니다.

- **방대한 생태계와 커뮤니티:** 오류가 발생하거나 모르는 것이 있을 때 Stack Overflow나 공식 문서를 통해 답을 찾는 것이 수월했습니다.
- **Hook(훅) 시스템:** `useState`, `useEffect` 등을 통해 직관적이고 깔끔하게 상태와 생명주기를 관리할 수 있습니다.

## 2. 왜 Next.js 인가?

React만으로도 훌륭한 앱을 만들 수 있지만, 블로그나 서비스 배포를 준비하면서 몇 가지 한계에 부딪혔습니다. 특히 검색 엔진 최적화(SEO)와 초기 로딩 속도 문제가 컸습니다.

### SSR (Server-Side Rendering) 과 SEO
블로그는 구글이나 네이버 등 검색 엔진에 내 글이 잘 노출되는 것이 생명입니다. React의 기본 방식인 CSR(Client-Side Rendering)은 크롤러 봇이 내용을 읽기 어렵다는 단점이 있습니다. Next.js는 SSR과 SSG(Static Site Generation)를 지원하여 페이지 로딩 전 HTML이 완성된 채로 서빙되기 때문에 검색 엔진 최적화에 탁월했습니다.

### 직관적인 라우팅 시스템 (App Router)
React에서 페이지를 이동하려면 `react-router-dom` 같은 추가 라이브러리를 설치하고 세팅해야 했지만, Next.js의 App Router 방식은 `app` 디렉토리 아래에 폴더를 만들기만 하면 자동으로 경로가 설정됩니다. 

```tsx
// app/blog/page.tsx 예시
import Link from 'next/link';

export default function BlogPage() {
  return (
    <main>
      <h1>나의 블로그 포스트들</h1>
      <Link href="/blog/hello-world">첫 번째 글로 이동</Link>
    </main>
  );
}
```
*이처럼 폴더 구조가 곧 웹사이트의 URL 구조가 되는 파일 기반 라우팅은 개발 시간을 크게 단축시켰습니다.*

### 뛰어난 개발자 경험 (DX)
Vercel 플랫폼과의 연동을 통해 Github에 Push만 하면 자동으로 빌드되고 배포되는 과정은 환상적이었습니다. 이 외에도 이미지 최적화(`<Image />`), 폰트 최적화 등 자체적으로 성능을 튜닝해 주는 기능 덕분에 서비스 퀄리티를 쉽게 높일 수 있었습니다.

## 앞으로의 다짐 🚀

프레임워크는 도구일 뿐, 결국 중요한 것은 자바스크립트에 대한 근본적인 이해와 좋은 설계를 해내는 능력이라고 생각합니다. Next.js를 열심히 활용하여 나만의 블로그를 멋지게 리팩토링해 나가고, 동시에 TypeScript와 Tailwind CSS 같은 트렌디한 기술들도 함께 학습하여 훌륭한 프론트엔드 개발자로 성장하고 싶습니다.
